# 前后端接口对接单 v2.1

> **用途**：前后端双方（及各自的 AI 助手）以此文档为唯一标准。任何一方修改接口格式，必须先更新本文档并通知对方。
>
> **当前版本**：基础版 v2.1（含影火AI DeepSeek V4，15 种消息类型全覆盖）
>
> **后端实现**：`后端/main.py`
>
> **前端参考**：`前端/AI影伴系统_v1.0_后端服务器联调接口文档.md`（同一份协议，两端视角不同）

## 一、连接信息

| 项目 | 值 |
|------|----|
| 协议 | WebSocket（`ws://`，非加密） |
| 端口 | **8000** |
| 后端 IP | 由后端同学启动服务器时告知（同 WiFi 下通过 `ipconfig` 查看 IPv4 地址） |
| URL 格式 | `ws://<IP>:8000/ws?room=<房间号>&role=<patient\|guardian>` |

**示例**（假设后端 IP 为 `10.39.86.240`）：
```
ws://10.39.86.240:8000/ws?room=TEST001&role=patient
ws://10.39.86.240:8000/ws?room=TEST001&role=guardian
```

**参数说明**：
- `room`：房间号，字符串，患者和监护人必须用同一个房间号才能配对。建议测试时用 `TEST001`
- `role`：身份标识，只能是 `patient`（患者）或 `guardian`（监护人），**这是 URL 参数，不是 JSON 字段**

## 二、房间与角色机制

### 2.1 角色如何确定

角色由 WebSocket **连接时的 URL 参数** `role` 决定，不需要在 JSON 消息里额外传 `role` 字段。

```
前端 App 的行为：
  用户选"我是患者" → 连接 ws://IP:8000/ws?room=xxx&role=patient
  用户选"我是监护人" → 连接 ws://IP:8000/ws?room=xxx&role=guardian
```

### 2.2 房间配对逻辑

1. 患者先连 → 房间状态为 `waiting`（等待监护人）
2. 监护人连入 → 房间状态变为 `paired`（已配对），双方都收到通知
3. 任一方断开 → 对方收到 `room_status` 通知，状态变回 `waiting`
4. 双方都断开 → 房间自动销毁（不保留任何数据）

## 三、消息格式规范

### 3.1 统一外壳

**所有** WebSocket 消息都使用以下 JSON 格式：

```json
{
  "msg_id": "字符串，UUID v4 格式，每条消息唯一",
  "type": "字符串，消息类型（见3.2表格）",
  "data": { /* 对象，具体结构随 type 不同而变化 */ },
  "timestamp": "字符串，ISO 8601 格式，如 2026-07-10T14:30:00"
}
```

---

### 3.2 全部消息类型（v2.1，共 15 种）

#### 说明
- **方向**列：`→` 表示"发送到"
- 转发类消息（⚡标记），服务器收到后原封不动发给房间内另一个人
- ★ 标记为 v2.1 新增

#### 消息列表

##### 心跳类

| type | 方向 | 用途 | data 结构 |
|------|------|------|-----------|
| `ping` | App→Server | 心跳保活（每 20 秒） | `{"heartbeat": "ping"}` |
| `pong` | Server→App | 心跳回复 | `{}`（空对象） |

##### 房间类

| type | 方向 | 用途 | data 结构 |
|------|------|------|-----------|
| `room_status` | Server→App | 服务器告知房间状态 | `{"room":"xxx","status":"waiting\|paired","your_role":"xxx","peer_role":"xxx","other_user":"xxx"}` |

##### 推流类

| type | 方向 | 用途 | data 结构 |
|------|------|------|-----------|
| `stream_start` | 患者→Server | 通知开始视频/音频推流 | `{"stream_type":"video\|audio"}` |
| `stream_url` | Server→监护人 | 下发推流播放地址 | `{"url":"rtmp://...","stream_type":"video\|audio","status":"live"}` |

##### 数据转发类（⚡ 服务器原封不动转发）

| type | 方向 | 用途 | data 结构 |
|------|------|------|-----------|
| `location_update` | ⚡ 患者→监护人 | GPS 位置同步 | `{"lat":39.9042,"lng":116.4074,"accuracy":10.0,"is_privacy_mode":false,"is_sos":false}` |
| `device_status` | ⚡ 患者→监护人 | 设备开关状态变化 | `{"camera":true,"headphone":true,"microphone":false,"battery":85,"network_type":"wifi"}` |
| `manual_message` | ⚡ 双向 | 手动发文字消息 | `{"content":"消息内容","from_role":"patient\|guardian"}` |
| `device_control_request` | ⚡ 双向 | 远程请求开关设备 | `{"device":"camera\|headphone\|microphone","action":"on\|off","from_role":"guardian"}` |

##### 紧急求助类

| type | 方向 | 用途 | data 结构 |
|------|------|------|-----------|
| `sos_alert` | ⚡ 患者→监护人 | 紧急求助 | `{"lat":39.9042,"lng":116.4074,"message":"患者触发SOS","stream_url":"","is_auto_detected":false}` |

> ⚠️ **服务器特殊处理**：收到 `sos_alert` 后，如果 `stream_url` 为空，服务器自动填入推流地址，再转发给监护人。

##### 通话信令类（⚡ 原封不动转发）

| type | 方向 | 用途 | data 结构 |
|------|------|------|-----------|
| `call_request` | ⚡ 双向 | 发起通话请求 | `{"call_type":"video\|audio","from_role":"patient\|guardian"}` |
| `call_response` | ⚡ 双向 | 通话应答 | `{"accepted":true\|false,"from_role":"patient\|guardian"}` |

##### AI 类（基础版预留，暂时作为转发通道供测试）

| type | 方向 | 用途 | data 结构 |
|------|------|------|-----------|
| `ai_voice_command` | ⚡ Server→患者 | AI 语音指令（患者端 TTS 播报） | `{"voice_text":"请注意前方障碍物","action_code":"play_warning_sound","priority":1}` |
| `danger_detected` | ⚡ Server→监护人 | AI 检测到危险 | `{"danger_type":"ai_detected","confidence":0.85,"message":"影火AI检测到潜在风险","patient_text":"..."}` |

> 📌 **基础版说明**：`ai_voice_command` 和 `danger_detected` 在完整版中由 AI 服务器生成。现在 v2.1 中 `danger_detected` 由影火AI 自动生成并推送给监护人。

##### ★ 影火AI 类（v2.1 新增）

| type | 方向 | 用途 | data 结构 |
|------|------|------|-----------|
| `chat_ai_request` | ★ 患者→Server | 向影火AI 发送消息 | `{"text":"用户说的话","context":{"lat":39.9,"lng":116.4,"time":"14:30","emotion":"平静"}}` |
| `chat_ai_response` | ★ Server→患者 | 影火AI 回复 | `{"reply":"AI回复的文字","is_danger":false}` |

> **`chat_ai_request` 处理流程**：
> 1. 服务器收到 → 调用 DeepSeek V4 API（附带影火AI system prompt + 上下文）
> 2. 返回 `chat_ai_response` 给患者 → 前端 TTS 朗读 + UI 显示
> 3. 如果 AI 回复中含 `[RISK]` 标记 → 服务器**同时**向监护人发 `danger_detected`

### 3.3 关键消息的详细示例

**① 患者连接 → 服务器回复 room_status（等待中）：**
```json
{
  "msg_id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
  "type": "room_status",
  "data": {
    "room": "TEST001",
    "status": "waiting",
    "your_role": "patient",
    "peer_role": null,
    "other_user": "等待中..."
  },
  "timestamp": "2026-07-10T14:30:00"
}
```

**② 监护人连接 → 双方都收到 room_status（已配对）：**
```json
{
  "msg_id": "b2c3d4e5-f6a7-8901-bcde-f12345678901",
  "type": "room_status",
  "data": {
    "room": "TEST001",
    "status": "paired",
    "your_role": "patient",
    "peer_role": "guardian",
    "other_user": "已连接"
  },
  "timestamp": "2026-07-10T14:30:05"
}
```

**③ 患者发位置 → 服务器转发给监护人（原封不动）：**
```json
{
  "msg_id": "loc-001",
  "type": "location_update",
  "data": {"lat": 39.9042, "lng": 116.4074, "accuracy": 10.0, "is_privacy_mode": false, "is_sos": false},
  "timestamp": "2026-07-10T14:30:10"
}
```

**④ 监护人发文字消息 → 服务器转发给患者：**
```json
{
  "msg_id": "msg-001",
  "type": "manual_message",
  "data": {"content": "前方有台阶，请注意", "from_role": "guardian"},
  "timestamp": "2026-07-10T14:30:15"
}
```

**⑤ 患者触发 SOS → 服务器填 stream_url 后转发给监护人：**
```json
{
  "msg_id": "sos-001",
  "type": "sos_alert",
  "data": {
    "lat": 39.9042, "lng": 116.4074,
    "message": "患者手动触发紧急求助！",
    "stream_url": "rtmp://localhost/live/TEST001",
    "is_auto_detected": false
  },
  "timestamp": "2026-07-10T14:30:20"
}
```

## 五、服务器转发规则总表

| 收到消息类型 | 处理方式 |
|-------------|---------|
| `ping` | 服务器回复 `pong`，不转发 |
| `room_status` | 服务器自己生成，不接收 |
| `stream_url` | 服务器自己生成，不接收 |
| `pong` | 服务器自己生成，不接收 |
| `location_update` | **原封不动**转发给同房间对方 |
| `device_status` | **原封不动**转发给同房间对方 |
| `manual_message` | **原封不动**转发给同房间对方 |
| `device_control_request` | **原封不动**转发给同房间对方 |
| `stream_start` | 服务器**生成新 `stream_url` 消息**发给对方 |
| `sos_alert` | 补充 `stream_url`（如为空）后转发给对方 |
| `call_request` | **原封不动**转发给同房间对方 |
| `call_response` | **原封不动**转发给同房间对方 |
| `ai_voice_command` | **原封不动**转发给同房间对方（基础版供测试） |
| `danger_detected` | **原封不动**转发给同房间对方（基础版供测试） |
| 未知类型 | 回复 `error` 消息 |
| `chat_ai_request` | ★ 调用 DeepSeek V4 API，返回 `chat_ai_response` 给患者；检测到风险时同时发 `danger_detected` 给监护人 |
| `chat_ai_response` | 服务器自己生成，不接收 |

```json
{
  "msg_id": "...",
  "type": "error",
  "data": {"message": "未知消息类型: xxxx"},
  "timestamp": "..."
}
```

## 八、快速参考卡片

**前端 AI 助手请按以下规则生成代码：**

1. 连接地址：`ws://<IP>:8000/ws?room=<房间号>&role=<patient|guardian>`
2. 每条消息必须包含四个字段：`msg_id`（UUID）、`type`（字符串）、`data`（对象）、`timestamp`（ISO 8601）
3. 连接后立刻收到 `room_status`，根据 `status` 字段判断配对状态
4. `location_update`、`device_status`、`manual_message`、`device_control_request`、`call_request`、`call_response`、`ai_voice_command`、`danger_detected` 类型消息会被服务器自动转发
5. `sos_alert` 会被服务器补充 `stream_url` 后转发
6. `stream_start` 会触发服务器生成 `stream_url` 发给监护人
7. 发送 `ping` → 收到 `pong`，用于心跳检测
8. 收到 `error` 类型消息 → 打印日志
9. ★ 发送 `chat_ai_request`（`{"text":"...", "context":{...}}`) → 收到 `chat_ai_response`（`{"reply":"...", "is_danger":false}`)
10. ★ `chat_ai_response.reply` 自动触发 TTS 朗读

**后端 AI 助手请按以下规则生成/验证代码：**

1. 端口 `8000`，角色从 URL 参数 `role` 读取，必须校验只允许 `patient` 或 `guardian`
2. 接收 `ping` 回复 `pong`
3. `room_status` 需包含 `room`、`status`、`your_role`、`peer_role`、`other_user` 字段
4. `stream_start` → 生成 `stream_url` 发给对方
5. `sos_alert` → 补充空 `stream_url` 后转发给对方
6. 其余类型消息（见转发规则总表）原封不动转发给房间内另一个人
7. 有人断开时通知对方，双方都断开时清理房间
8. ★ `chat_ai_request` → 调用 DeepSeek V4 API（system prompt 见 main.py） → 返回 `chat_ai_response`；AI 标记 [RISK] 时 → 同时发 `danger_detected` 给监护人
